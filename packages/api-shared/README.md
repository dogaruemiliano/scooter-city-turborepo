# `@repo/api-shared`

Hand-written, zero-build types and constants shared between every consumer of the API: `apps/api`, `apps/web`, and `apps/mobile` (once it lands). Anything Orval cannot generate from the OpenAPI document lives here.

## What lives here vs in `@repo/api-generated`

| Concern                                                                         | Source                                                   | Used by                                                                                                                                |
| ------------------------------------------------------------------------------- | -------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| Request / response DTOs and operation functions                                 | `@repo/api-generated` (Orval, regenerated by `pnpm gen`) | api, web, mobile                                                                                                                       |
| Cookie names (`access_token`, `refresh_token`)                                  | **here**                                                 | api (sets), web `proxy.ts` (reads). Mobile uses Bearer tokens instead, but importing the constants is harmless.                        |
| `SessionUser`, `SessionSummary`, `EnabledAuthMethods`, `TokenPair` types        | **here**                                                 | api (return shapes), web (React context), mobile (auth context) — these wrap the wire-level types in a stable, single name across apps |
| Zod fragments (`emailSchema`, `phoneSchema`, `otpCodeSchema`, `passwordSchema`) | **here**                                                 | api DTOs, web server-action input validation, mobile form validation — all three sides reuse the same rules                            |
| `ROUTES` — fully-qualified path constants for every auth endpoint               | **here**                                                 | api (smoke tests), web (`proxy.ts` decides what to refresh on), mobile (manual `fetch` paths where Orval output is awkward)            |

The split is deliberate: regenerating `@repo/api-generated` must never trample hand-maintained code. If something is auto-generated from OpenAPI, it lives there. If it is hand-maintained, it lives here.

## Adding something to this package

1. Add it under `src/<concern>.ts`.
2. Re-export from `src/index.ts` in the matching section.
3. Add a unit test under `test/` if the export has runtime behavior (zod refines, regex, etc.).
4. Update this README's table above if the new export is a load-bearing concept.

## Usage

```ts
// any of: apps/api, apps/web, apps/mobile
import {
  ROUTES,
  ACCESS_TOKEN_COOKIE,
  passwordSchema,
  type SessionUser,
} from "@repo/api-shared";

const path = ROUTES.credentials.login; // "/v1/auth/credentials/login"
```

## Why no build step

This package is consumed only inside the workspace. Every consumer transpiles TS itself (Nest for the API; Next for the web; Metro / EAS for mobile). Skipping `tsc` here keeps `pnpm dev` startup quick and removes a class of stale-build bugs.
